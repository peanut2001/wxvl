青龙面板作为一款常用的自动化任务管理工具，因鉴权逻辑设计缺陷存在严重的大小写绕过漏洞，攻击者可通过构造特殊路径绕过身份验证，实现账号接管、远程代码执行、任意文件上传 / 读写等高危操作，对服务器安全造成极大威胁。本文将从漏洞环境搭建、实际漏洞复现、底层原理剖析三个维度，完整还原该漏洞的利用过程，并深入解析漏洞产生的核心原因与危害延伸。    
## 一、环境搭建  
  
本次复现基于青龙面板 2.20.1-debian 版本，采用 Docker Compose 快速搭建漏洞环境，步骤如下：    
  
创建文件夹后进入文件夹，创建docker-compose.yml文件  
```
version: '3'
services:
  qinglong:
    image: whyour/qinglong:2.20.1-debian
    container_name: qinglong
    hostname: qinglong
    restart: unless-stopped
    ports:
      - "5700:5700"
    volumes:
      - ./data:/ql/data
    environment:
      - QlBaseUrl=/
    networks:
      - qinglong_net
networks:
  qinglong_net:
    driver: bridge
```

## 二 漏洞实际复现    
  
该漏洞核心为**路径大小写鉴权绕过**  
，利用 Express 框架路由匹配不区分大小写、而青龙面板鉴权逻辑对路径大小写严格校验的矛盾点，构造/Open/  
、/API/  
等混合大小写路径，即可跳过身份验证，直接访问敏感接口。以下为核心高危漏洞的复现过程：  
  
1、未授权重置管理员密码（账号接管）      
  
构造 PUT 请求，访问/Open/system/auth/reset  
接口；请求体携带需修改的账号密码：{"username":"admin","password":"123abc"}  
；发送请求后返回code:200  
与更新成功  
提示，密码重置完成；后续可通过该接口获取有效 token，直接以管理员身份登录面板。

发送请求后得到token  
  
![](https://mmbiz.qpic.cn/mmbiz_png/2AAMh9HmvsRLLIBCOYF6wRUbO3qeX431QSy6iaq7kLmbJxFUhFcBib3MM2ICFyFbjYRAgVZFiaWfG2O10FJFtAksNe5h234wL95iajgFrpFVxEo/640?wx_fmt=png&from=appmsg "")  
  
2、未授权写入文件  
  
利用脚本执行接口，向服务器写入任意文件，可植入恶意脚本为后续代码执行做铺垫。构造 PUT 请求，访问/Api/scripts/run  
接口；请求体指定文件名与内容：{"filename":"log.sh","content":"#!/bin/bash\ndate>/tmp/log.txt"}  
；发送请求后返回code:200  
，表示文件写入并执行成功；进入服务器终端，执行cat /tmp/log.txt  
，可查看到文件已成功生成并写入内容。

3、未授权远程命令执行  
  
通过系统命令执行接口，直接向服务器发送任意 Shell 命令，实现无文件落地的远程代码执行，危害最大。构造 PUT 请求，访问/0pen/system/command-run  
接口；请求体携带需执行的命令：{"command":"ls"}  
；发送请求后，服务器直接返回命令执行结果，包含面板目录下的所有文件，如notify.py  
、ql_sample.js  
等；攻击者可替换命令为反弹 Shell、文件删除等高危操作，完全控制服务器。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/2AAMh9HmvsRGjOcON4BaPSr4iaY9V3S3DLtLEcxJh1y0oDzfusZC6b3Gok0rvcEdvtCzRicebcLVyz3rzeJdaPuQSemiaZ1NWPLF395cmkKm9s/640?wx_fmt=png&from=appmsg "")

 正常访问/api/user/avatar  
上传接口，删除 token 后会返回401 未授权  
，无法上传；

 正常访问/open/system/data/export

## 三 总结  
  
本次青龙面板鉴权绕过漏洞是典型的**基础逻辑设计缺陷**  
导致的高危安全问题，核心问题在于开发人员忽略了 Express 框架路由匹配的特性，在鉴权层使用大小写敏感的校验方法，再加上业务层无任何二次验证、参数校验，最终导致漏洞危害被无限放大。
