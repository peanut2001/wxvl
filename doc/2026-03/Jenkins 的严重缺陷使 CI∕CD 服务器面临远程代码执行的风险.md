#  Jenkins 的严重缺陷使 CI/CD 服务器面临远程代码执行的风险  
sec随谈
                    sec随谈  sec随谈   2026-03-20 00:48  
  
Jenkins 项目发布了一份重要的安全公告，指出存在多个可能导致系统完全被攻破的**漏洞**。该公告重点提及 Jenkins 核心代码中的两个高危漏洞，以及一个常用第三方插件中的凭证处理问题。  
  
作为全球 CI/CD 流水线的支柱，这些 Jenkins 漏洞对 DevOps 团队构成重大风险，攻击者可能借此将恶意代码直接注入软件开发生命周期的核心。  
  
最直接的威胁涉及一个“高”严重性任意文件创建**漏洞**（CVE-2026-33001）。Jenkins 2.554 及更早版本，以及 LTS 2.541.2 及更早版本，在提取 .tar 或 .tar.gz 归档文件时，无法安全地处理符号链接。  
> “这使得精心设计的归档文件能够将文件写入文件系统上的任意位置，仅受运行 Jenkins 的用户的文件系统访问权限的限制。”  
  
  
对于从控制器提取的归档文件，其影响十分严重。攻击者可以通过“向 JENKINS_HOME/init.groovy.d/ 目录写入恶意脚本”或部署恶意插件来实现远程代码执行 (RCE)。此漏洞尤其危险，因为任何拥有“Item/Configure”权限或能够控制代理进程的用户都可以触发此漏洞。  
  
第二个“高危”**漏洞**（CVE-2026-33002）针对的是Jenkins命令行界面（CLI），当通过WebSocket访问时存在安全隐患。Jenkins被发现使用不安全的HTTP请求头来验证这些连接的来源，这使得DNS重绑定攻击有机可乘。  
> “攻击者通过诱使受害者访问恶意网站，该网站使用 DNS 重绑定解析到 Jenkins 控制器的 IP 地址，从而可以从不受信任的来源建立到 CLI 端点的 WebSocket 连接，并以匿名用户的身份执行 CLI 命令。”  
  
  
如果匿名用户被授予权限，或者服务器采用“任何人都可以做任何事”的策略，攻击者可以使用 Groovy 脚本命令执行任意代码。  
  
除了核心修复之外，LoadNinja 插件（v2.1 及更早版本）还被发现存在泄露敏感数据的问题（CVE-2026-33003、CVE-2026-33004）。该插件“将 LoadNinja API 密钥以未加密的形式存储在作业配置的 config.xml 文件中”，并且未能在 Web 界面中对其进行加密处理。任何拥有“项目/扩展读取”权限的用户都可以轻松获取这些密钥。  
  
强烈建议管理员立即升级实例。Jenkins 已发布以下补丁来修复这些漏洞：  
- Jenkins Weekly：更新至 2.555 版本。  
- Jenkins LTS：更新至版本 2.541.3。  
- LoadNinja插件：更新至2.2版本。  
如果无法立即更新 CLI 漏洞，管理员应“为其 Jenkins 控制器设置身份验证，并删除匿名用户的权限”。
