Start  
  
    前段时间圈子里发过一个群杰印章的反序列化漏洞  
  
![](https://mmbiz.qpic.cn/mmbiz_png/I8PshYBGmQ7ugUzicE3IaoHvlpzmF58vytYS5q04H4yXkmfSdFibYNicMCicOWXrcbCFCvLpzpmhIO9GufzNLwYBBa3mBricFAHmabJ4lSeHYCTY/640?wx_fmt=png&from=appmsg "")  
  
      
该漏洞的核心利用逻辑在于，通过 userNameLogin 接口获取具备后台接口调用权限的 Token。但随着厂商持续修复，越来越多站点在访问该接口时返回 404，导致后续利用链路受阻  
。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/I8PshYBGmQ6vMt4B08VBorku4voO1FN14TA8BAnW6YDhD8HD5xJnrLvhheUhv5PibbeljmXUxUR6mz7T0ONQnHPoEBnUcPJInrfsLVYWsSVo/640?wx_fmt=png&from=appmsg "")  
  
  
2  
  
Action  
  
    偶然的机会下，发现存在漏洞的A系统与B系统的token竟然可以通用？也就是A系统的token放在B系统也是有效的，B系统的token放在A系统也是有效的！感觉这个系统的token生成方式有点说法的？  
  
  
    剖析一下源代码  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/I8PshYBGmQ75MEia8J1icJicibK5SzYlJU75K2bnl8BHLfxExoUseJ0brRABeBloibXsZxQNDY7Spe8d3CSia2xh5uhwUBiaANwUcSH8F9jZsGOwcM/640?wx_fmt=png&from=appmsg "")  
  
    token的生成逻辑主要在这个getNewToken里面，其实就是通过rsa加密的方式将一些信息加密了一下，不同系统之间能够通用token估计也就是校验token有效性的时候仅校验了例如用户名、时间戳之类的信息  
  
![](https://mmbiz.qpic.cn/mmbiz_png/I8PshYBGmQ4iax6xHLwfHDgwkPnSHlKavahianfRG0b3423iaQeFSQ9laqsZfEhrxWpibGAvqeoN22X5BCczY8qr2TxPkpQiaicUguHJibzQ6fwc0w/640?wx_fmt=png&from=appmsg "")  
  
    思路清晰了，那后面的内容就可以交给AI了，让ai根据要求生成能直接生成token的脚本即可。这样，即使有些站点没有获取token的接口，也能直接通过脚本生成的token进行后续链路的使用。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/I8PshYBGmQ694TKdLYRj4cvUoNU6UI8hFAZQExT09yArxI0m2VJFia2avJic0xKnK3mDica7LwB9J4FCmGV1gHTkLJ2ia2MhuSGBicsDLFibPrMTM/640?wx_fmt=png&from=appmsg "")  
  
    最后，最新版本下该方式貌似不能用，用于判断的方式为访问  
/web/userNameLogin接口，服务端返回  
 405，能打的返回404。  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/I8PshYBGmQ5nBE7IQ1BppKTMVMyE3eaYpFPQ32IP0IfBsenOOvnUWDK8q885OQdTKF9h95cXEHuWqEDqrH6l34xjwThsDWHkZAY7FU2QX10/640?wx_fmt=png&from=appmsg "")
