Google 威胁情报组 (GTIG) 发现了一套强大的全新漏洞利用工具包，其攻击目标涵盖运行 iOS 13.0 (2019 年 9 月发布) 至 iOS 17.2.1 (2023 年 12 月发布) 的各型号 Apple iPhone 设备。该工具包被开发者内部命名为 "Coruna"，包含 5 条完整的 iOS 漏洞利用链，共计 23 个漏洞利用程序。其核心技术价值在于全面的 iOS 漏洞利用集合，其中最为精密的部分采用了未公开的利用技术和缓解措施绕过手法。  
  
Coruna 工具包提供了 高级攻击能力扩散趋势的又一例证。2025 年全年，GTIG 持续追踪其使用轨迹：最初在某 监控供应商 客户实施的高度定向攻击行动中被捕获，随后发现疑似俄罗斯间谍组织 UNC6353 将其部署于针对乌克兰用户的水坑攻击中。此后，当来自中国的经济利益驱动型威胁行为者 UNC6691 在大规模攻击活动中使用该工具包时，我们获取了其完整副本。扩散的具体途径尚不明确，但此现象暗示着一个活跃的 "二手" 零日漏洞利用市场的存在。除已识别的漏洞利用外，多个威胁行为者已掌握可复用和改造的高级利用技术，并能将其与新发现的漏洞结合使用。  
  
依据我们的 披露政策，我们公开分享此项研究以提高安全意识并推动行业整体安全水平的提升。我们已将所有已识别的网站和域名纳入 Safe Browsing，以保护用户免受进一步侵害。Coruna 工具包对最新版本 iOS 已无效，我们强烈建议 iPhone 用户将设备更新至 iOS 最新版本。若无法更新，建议启用 锁定模式 (Lockdown Mode) 以增强安全防护。

### 初始发现：商业监控供应商的角色  
  
2025 年 2 月，我们捕获到某监控公司客户所使用的部分 iOS 漏洞利用链。这些漏洞利用被整合进一个此前从未出现过的 JavaScript 框架中，该框架采用了简单但独特的 JavaScript 混淆技术。  
```
[16, 22, 0, 69, 22, 17, 23, 12, 6, 17].map(x => {return String.fromCharCode(x ^ 101);}).join("")
```  
```
i.p1=(1111970405 ^ 1111966034);
```  
  
该 JavaScript 框架利用上述构造对字符串和整数进行编码。  
  
框架首先启动指纹采集模块，收集多种数据点来判断目标设备是否为真实设备，并确定其具体的 iPhone 型号和 iOS 软件版本。基于采集到的数据，框架加载对应的 WebKit 远程代码执行 (RCE) 漏洞利用，随后加载指针认证码 (PAC) 绕过，如图 2 中反混淆后的 JavaScript 所示。

2025 年夏季，我们发现同一 JavaScript 框架被托管于 cdn.uacounter  
.  
com，该站点以隐藏 iFrame 的形式嵌入了大量被入侵的乌克兰网站中，涵盖工业设备、零售工具、本地服务及电子商务等多种类型的网站。该框架仅向来自特定地理位置的选定 iPhone 用户投放。  
  
该框架与此前发现的完全一致，投放的漏洞利用集合也相同。在服务器关闭前，我们收集到了其中的 WebKit RCE 漏洞利用，包括 CVE-2024-23222、CVE-2022-48503 和 CVE-2023-43000。我们随即向 CERT-UA 发出警报并与其合作清理了所有被入侵的网站。

我们获取了所有经过混淆处理的漏洞利用程序，包括最终载荷。在进一步分析中，我们发现攻击者在某个实例中部署了漏洞利用工具包的调试版本，所有漏洞利用程序及其内部代号均以明文形式暴露。正是在此时，我们得知该工具包的内部名称可能为 Coruna。我们总共收集了数百个样本，涵盖 5 条完整的 iOS 漏洞利用链。该工具包能够攻击运行 iOS 13.0 (2019 年 9 月发布) 至 iOS 17.2.1 (2023 年 12 月发布) 的各型号 iPhone 设备。  
  
在后续章节中，我们将简要介绍该框架、分析各漏洞利用链的构成，以及我们所捕获的相关植入体。我们对已收集数据的分析仍在进行中，预计将通过新的博客文章或 根因分析 (RCA) 发布更多技术细节。  
### Coruna 漏洞利用工具包  
  
漏洞利用工具包外围框架的工程设计极为精湛；各漏洞利用组件衔接自然，通过通用的工具库和利用框架有机整合。该工具包执行以下独特操作：  
- 若设备处于锁定模式 (Lockdown Mode) 或用户正在使用无痕浏览，则自动退出。  
  
- 整个过程使用唯一的硬编码 cookie 来生成资源 URL。  
  
- 资源通过哈希值引用，需使用唯一 cookie 按 sha256(COOKIE + ID)[:40]  
进行派生以获取对应 URL。  
  
- RCE 和 PAC 绕过以未加密形式投放。  
  
该工具包包含一个二进制加载器，用于在 WebKit 内实现 RCE 后加载对应的漏洞利用链。此场景中，二进制载荷具有以下特征：  
- 包含唯一的元数据，标识其真实类型以及支持的芯片和 iOS 版本。  
  
- 通过以 .min.js 结尾的 URL 提供。  
  
- 使用 ChaCha20 加密，每个 blob 拥有唯一密钥。  
  
- 以自定义文件格式封装，文件头为 0xf00dbeef。  
  
- 使用 Lempel-Ziv-Welch (LZW) 算法压缩。

### 最终载荷  
  
在漏洞利用链的最后阶段，一个名为 PlasmaLoader  
的阶段加载器二进制文件 (GTIG 将其追踪为 PLASMAGRID) 以 com.apple.assistd 为标识，负责与漏洞利用所建立的内核组件进行通信。该加载器将自身注入 powerd——一个在 iOS 上以 root 权限运行的守护进程。  
  
注入的载荷并不具备我们通常在监控供应商产品中预期的功能，而是专注于窃取金融信息。该载荷能够解码磁盘上图像中的二维码，还配备了一个文本分析模块，用于检索 BIP39 助记词序列或特定关键词 (如 "backup phrase" 或 "bank account")。若在 Apple 备忘录中发现此类文本，相关内容将被回传至 C2 服务器。  
  
更重要的是，该载荷具备远程收集和执行额外模块的能力，配置信息从 http://<C2 URL>/details/show.html  
获取。配置文件及额外模块均以 7-ZIP 压缩包形式存储，并使用唯一的硬编码密码进行保护。配置以 JSON 格式编码，内容为模块名称及其对应 URL、哈希值和大小的列表。  
```
{  "entries": [    {      "bundleId": "com.bitkeep.os",      "url": "http://<C2URL>/details/f6lib.js",      "sha256": "6eafd742f58db21fbaf5fd7636e6653446df04b4a5c9bca9104e5dfad34f547c",      "size": 256832,      "flags": {        "do_not_close_after_run": true      }    }...  ]}
```  
  
如预期，几乎所有已识别的模块都采用统一设计；它们均通过放置函数钩子来窃取以下应用的加密货币钱包或敏感信息：

所有这些模块都包含规范的日志记录，日志语句以中文编写：  
```
<PlasmaLogger> %s[%d]: CorePayload 管理器初始化成功，尝试启动...
```  
  
该日志字符串表明 CorePayload 管理器初始化成功  
  
部分注释还包含 emoji 表情，其写作风格暗示它们可能由 LLM 生成。  
```
<PlasmaLogger> %s[%d]: [PLCoreHeartbeatMonitor] ✅ 心跳监控已启动 (端口=0x%x)，等待 CorePayload 发送第一个心跳...
```  
  
网络通信通过 HTTPS 进行，收集的数据经 AES 加密后以 POST 方式发送，密钥为某静态字符串的 SHA256 哈希值。部分 HTTP 请求包含额外的 HTTP 头字段，如 sdkv  
或 x-ts  
，后跟时间戳。植入体包含一组硬编码的 C2 地址，并设有备用机制以应对服务器无响应的情况。植入体内嵌了一个自定义域名生成算法 (DGA)，以字符串 "lazarus" 为种子生成一系列可预测的域名。这些域名长度为 15 个字符，使用 .xyz 作为顶级域名。攻击者使用 Google 的公共 DNS 解析器来验证域名是否处于活跃状态。
