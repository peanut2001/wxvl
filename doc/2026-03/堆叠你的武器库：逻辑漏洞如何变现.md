##   
> 什么是业务逻辑漏洞？  
> 业务逻辑漏洞是应用程序设计或实现中的缺陷，它允许攻击者触发非预期行为。这可能让攻击者操纵合法功能以达到恶意目的。此类漏洞通常是由于未能预见应用程序可能出现的异常状态，从而未能安全地处理它们造成的。  
  
  
简而言之，逻辑漏洞通常意味着**绕过限制**  
。当应用告诉你“你不能这样做”时，这可能是逻辑漏洞存在的提示——允许你把**否**  
变成**是**  
。  
  
在本篇文章中，我将分享过去几个月猎取漏洞过程中发现的一些逻辑漏洞示例。这些示例简单，但对于新手猎人理解逻辑漏洞的查找方向非常有用。  
## 1. 绕过必填字段  
  
在一个提供 CRM 的外部应用中，我发现了“潜在客户（Leads）”部分的逻辑漏洞。每个潜在客户都有用户相关数据，其中 **Email**  
 字段被标记为必填  
。通常，如果你留空，应用会阻止保存。  
  
于是我想：**如果应用说“不能”，那就试着绕过它。**  
  
我先尝试在创建潜在客户时注入 HTML 标签 <script>  
。应用正确地编码了字符（<  
 → &lt;  
），所以没有问题。  
  
但是当我保存潜在客户后**再次编辑**  
时，发现了有趣的事情：  
  
在编辑时，如果在字段中插入 <script>  
，应用会**完全移除标签**  
并将字段**清空**  
，但仍允许我保存。这意味着我可以**完全绕过“必填字段”验证**  
。我写了报告并提交了。

## 2. 未授权使用“高级”功能  
  
在 CRM 中，应用有一个叫 **Contacts（联系人）**  
 的功能，用户可以存储联系人信息。我创建了两个账户：  
- 一个 **试用**  
账户  
  
- 一个 **免费**  
账户（通常无法访问 CRM）  
  
试用用户将免费用户邀请到自己的 CRM 工作区，我发现了有趣的事情：  
  
免费用户突然可以使用 **Contacts**  
 功能。  
  
随后，我用试用账户将免费用户从 CRM 中删除。此时，免费用户恢复到正常状态——**无 CRM 访问权限**  
。  
  
但是漏洞在这里：  
  
我检查 Burp 时发现，免费用户仍然可以发送请求来 **创建**  
、**编辑**  
和 **导出**  
联系人。  
  
尽管 UI 阻止了访问，但后台仍允许这些高级操作。  
  
也就是说，免费账户仅通过重放 API 请求就能使用**高级功能**  
。

## 3. 绕过页面数量上限  
  
你可能以为这是竞态条件——但**不是**  
 :)  
  
应用允许用户创建网站，页面数量有限制。  
  
对于免费用户，限制是 **50 页**  
。  
  
我尝试了常规竞态条件方法，但不起作用（看起来他们已修复）。  
  
我发现了一个更简单的绕过方法：  
  
用户可以删除页面，被删除的页面会进入 **垃圾箱**  
。从那里可以**恢复**  
它们。  
  
漏洞在这里：  
  
恢复已删除页面时，应用**没有检查**  
用户当前已有的活跃页面数量。  
  
我创建了 50 页（最大值），然后去垃圾箱恢复已删除页面。  
  
它们恢复**正常**  
，最终我拥有**超过 50 个活跃页面**  
。  
  
这是一个典型的后端验证缺失造成的逻辑漏洞。

## 4. 编辑已删除的产品  
  
在同一应用中，有创建商店和添加产品的功能。通常，将产品移入**垃圾箱**  
后，你**不应**  
再编辑它。必须先恢复才能编辑。  
  
但是“你不能” → **逻辑漏洞时间**  
。  
  
操作如下：  
1. 创建新产品，正常编辑，并在 Burp 捕获请求  
  
1. 删除产品（移入垃圾箱）  
  
1. 在 Burp 中重放 **编辑**  
 请求 → 请求仍然被接受  
  
所以我可以**在未恢复的情况下编辑已删除产品**  
——明显的逻辑漏洞。

## 5. 将页面移动到已删除文件夹  
  
在网站构建器中，用户可以创建 **文件夹**  
 并将页面移动其中。删除文件夹后，你**不应**  
再访问或编辑里面的内容。  
  
但“你不能”？**我的机会来了。**  
  
我注意到每个文件夹在 API 中有两个关键操作：  
1. **编辑文件夹**  
  
1. **将页面移动到文件夹**  
  
删除文件夹后，编辑操作会正确返回错误——无法访问。  
  
但是第二个操作呢？  
  
尝试 **“将页面移动到文件夹”**  
，服务器返回 **200 OK**  
 并成功执行。  
  
即使文件夹在垃圾箱中且 UI 无法访问，后台仍允许移动页面。  
  
这意味着用户可以**将页面隐藏在已删除的文件夹中**  
，从界面无法访问——另一种逻辑漏洞，由状态验证缺失造成。

## 6. 渲染失败导致整个演示崩溃  
  
在 HackerOne 上测试一个私有程序时，我发现了负责创建和展示幻灯片演示的逻辑漏洞。  
  
编辑器中可以更改演示背景色。编辑颜色的请求如下：  
```
```  
  
**RGB**  
 值表示红、绿、蓝分量。  
  
服务器使用此 RGB 值进行计算生成最终颜色。  
  
测试时，我将颜色值改为随机输入，发现服务器**接受了任何输入**  
。  
  
但是返回 UI 并尝试重新加载演示时，整个演示**崩溃**  
——没人可以访问。

原因：  
  
开发者完全信任客户端发送的值，直接将 RGB 字符串传入颜色计算函数，**未验证**  
值是否合法或格式正确。  
  
客户端发送的一个简单无效颜色值就导致渲染函数失败，从而整个演示对所有用户崩溃——明显的逻辑漏洞，来源于过度信任用户输入。  
## 7. 篡改日志  
  
在 Bugcrowd 上测试一个私有程序时，我发现了日志处理的逻辑漏洞。  
  
该应用严重依赖日志跟踪每个用户的操作。  
  
创建新字段时，我注意到数据库存储了一个 **created_by**  
 值。  
  
我拦截请求并将 **created_by**  
 邮箱改成任意用户。  
  
结果？  
  
请求被**接受**  
，系统成功在另一个用户邮箱下创建了操作。  
  
原因：  
  
开发者信任客户端请求的 **created_by**  
 参数。  
  
而不是使用认证 **session token**  
 来确定操作执行者，后台完全依赖用户提供的数据。  
  
这允许我篡改日志，让它看起来像 **任何用户**  
 执行了操作——明显的逻辑漏洞，由信任客户端控制参数造成。

## 8. 编辑组织内的预设角色  
  
创建新组织时，会有一些 **预设角色**  
（如 owner  
、reader  
 等）。  
  
用户也可以创建 **自定义角色**  
，并自由编辑。  
  
尝试编辑 **预设角色**  
 时，UI 会禁用编辑按钮并提示不可编辑。  
  
我操作如下：  
1. 创建自定义角色  
  
1. 打开编辑页面并拦截请求  
  
1. 将自定义角色 ID 替换为预设角色 ID  
  
结果：  
  
服务器**接受请求**  
，预设角色成功被编辑。  
  
逻辑漏洞显而易见：  
  
限制只在**客户端**  
执行，后台没有验证，未阻止编辑预设角色。
