今天开始，这边就开始给大家聊聊有关  
WebSockets的一些漏洞了。  
WebSocket 这玩意儿，说实话已经成了现代互联网的“水电煤”了。刷网页聊天、看股票实时行情、玩网页版小游戏，背后都是它在默默干活。  
  
![](https://mmbiz.qpic.cn/mmbiz_png/qg1MKHx3jGF7lfsvLnkFYadZoPGXl7LFvsNwaicqQicz1y02MD8qwicDwESSxXmdPSUlWpwkiaaRI2c0tUdDWHY4icyq61tiaKlLAGzJWUibLaicENA/640?wx_fmt=png&from=appmsg "")  
  
它厉害在哪？简单说就是：用 HTTP“敲门”建立连接，然后一脚踢开 HTTP，搞一条双向通道，服务器能随时主动给你推数据，不用你一遍遍去问“有新消息吗”。而且这通道一旦建好，就不轻易断开。  
  
但问题也来了——正因为   
WebSocket 处理的多是敏感数据和关键操作，那些年在传统 HTTP 上踩过的坑，什么越权、注入、会话劫持，几乎一个不落全都能在 WebSocket 上重演。  
  
所以今天咱就聊聊，怎么用   
Burp Suite 这套工具，去折腾、去试探、去挖掘   
WebSocket 里的那些安全漏洞。  
  
核心思路：让应用“没想到”  
  
挖掘 WebSocket 漏洞的思路，跟搞传统 Web 测试没啥本质区别——就是想方设法让程序处理一些它没预料到的情况。只不过 WebSocket 的“玩法”多了几层：  
- 改消息：拦截下来，篡改内容再放行  
  
- 造消息：把历史消息翻出来重放，或者自己编一条新的发出去  
  
- 控连接：连握手阶段都不放过，直接操纵连接本身  
  
  
Burp Suite 恰好把这三种玩法都安排得明明白白。  
  
第一关：拦截篡改，从“偷看”到“动手脚”  
  
先说最基础的——拦截 WebSocket 消息。  
  
操作路径很简单：  
  
打开 Burp 的内置浏览器，正常访问那些用到 WebSocket 的功能。怎么知道哪些功能用了？看   
Burp Proxy 的   
WebSocket 历史记录 标签页，有条目出现就说明有戏。  
  
切到工具的   
拦截 标签页，确保拦截开关是开着的。  
  
这时候，只要浏览器或服务器发出一条 WebSocket 消息，就会立刻停在拦截界面，原封不动展示给你看。你可以随便改里面的内容，改完了点   
转发，消息就会继续走人。  
  
有个小细节：你可以在 Burp 的设置里自定义拦截规则，比如只拦客户端发给服务器的，或者反过来只拦服务器下发的。按需配置就行。  
  
这一步的想象力空间其实挺大的——比如一个聊天应用，你拦截自己发的消息，把发送者   
ID 改成别人的，看看能不能  
冒充别人发言；或者一个数据看板，你拦截服务器返回的数据，把数字改大改小，看看前端会不会校验。  
  
第二关：重放与生成，让消息“反复去世”  
  
拦截只能改一次，如果想让同一条消息反复被服务器处理，或者想批量测试某个参数的边界情况，就得靠 Burp Repeater 了。  
  
操作也很简单：  
  
在 WebSocket 历史记录或者拦截界面，选中目标消息，右键   
发送到 Repeater。  
  
在   
Repeater 里，这条消息就任你摆布了——想怎么改就怎么改，改完点发送，服务器就得再处理一次。可以反复发送，直到你满意为止。  
  
不止能改已有的消息，你还能直接编一条新的，自己写内容，自己选方向（发往客户端还是服务器）。这手“凭空捏造”在某些场景下特别好用，比如测试服务器对畸形数据的处理能力。  
  
Repeater 里还有个   
历史记录 面板，这个设计挺贴心——它不仅记录了你手动发送的消息，还把这条连接上浏览器和服务器之间的所有往来消息都列出来了。相当于一个完整的会话存档。  
  
如果翻历史的时候发现某条消息值得再试试，直接选中它，右键 编辑并重新发送，就能把它捞出来继续折腾。  
  
这一步的价值在于“暴力试探”——比如一个购物网站的 WebSocket 接口，你用  
重放功能疯狂提交订单，看看有没有防重放机制；或者一个投票功能，你反复发送同一张票，看看会不会被计数多次。

第三关：操纵连接，从源头搞事情  
  
前面两招都是建立在连接已经建立的基础上。但有些场景，得从  
握手阶段就开始动手脚。  
  
为啥要动握手？原因如下：  
- 想探索更多攻击面：比如修改握手请求里的   
Origin 头，测试跨域限制是否到位  
  
- 连接意外中断：有些攻击可能导致   
WebSocket 掉线，需要快速重建  
  
- 凭证过期：握手请求里的   
token 可能有时效性，需要刷新后才能继续测试（源头就可以拿到实时的有效   
token）  
  
  
Burp Repeater 也考虑到了这些需求，操作方式挺直观：  
  
老规矩，先把一条   
WebSocket 消息发到   
Repeater。  
  
此时 WebSocket 功能页签中，会给你三个选项：  
- 附加到现有连接  
  
- 克隆一个已有连接  
  
- 重连一个已断开的连接  
  
  
如果选克隆或重连，向导会把完整的握手请求展示出来。这时候你就可以动手脚了——改 path、改 Host 头、更新过期的 token，想怎么改都行。  
  
改完了点   
连接，Burp 就会按照你定制的握手请求去尝试建立连接。如果成功了，你就可以用这个新连接继续发消息、做测试。  
  
这个功能在测试“连接复用”类的漏洞时特别好用。比如有些应用会在握手阶段校验身份，但建立连接后的消息就不再校验。这时候你就可以尝试克隆一个合法连接，然后用新连接发送越权消息，看看服务器会不会中招。  
  
最后：  
  
从  
改消息、  
造消息，到  
控连接，这三板斧基本覆盖了   
WebSocket 安全测试的大部分场景。思路其实不复杂，就是“让应用处理它没想到的情况”。只不过   
WebSocket 多了一层连接维度的玩法，需要多熟悉熟悉工具的操作。
