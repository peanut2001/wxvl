## 逆向工程  
  
一般来说，对此类物联网设备进行逆向工程的第一步是获取其软件，然后剖析它们以提取可执行二进制文件、配置文件等。  
  
设备上运行的固件可能来自不同来源，最易得的是从制造商服务器下载更新文件（OTA）。同时，也可以通过将实际设备的闪存芯片连接到闪存读取器硬件来直接读取其内容。根据启动加载程序（bootloader）的安全加固程度，U-Boot可能在编译时开启了交互式Shell（shell），这使得转储存储介质变得容易得多。  
  
有了系统分区的镜像文件后，就可以开始逆向工程中的静态分析部分了！  
  
大多数情况下，仅靠静态分析是不够的，还需要与运行在真实硬件上的设备进行交互。为了实现诸如重写配置或管理运行服务等目的，Shell访问权限至关重要。安全摄像头——它们本身就是嵌入式设备——通常有用于打印应用程序日志的串口（serial ports）。然而，有些串口也提供了登录Shell！也可能存在Telnet和SSH访问，这些可能被调试标志（debug flag）所限制。  
  
当逆向工程的系统范围缩小到单个可能存在漏洞的二进制文件时，就需要更高级的调试能力。这可以是一个静态编译的 gdb  
 服务器，部署到安全摄像头上，通过网络进行访问。  
## 小米C400  
  
在研究阶段，我要大声感谢我的同事Lorant Szabo的指导和帮助。  
  
这款摄像机面临的第一个挑战是物理拆解设备：其球罩部分由非常硬的卡扣将两部分固定在一起。  
  
在其SigmaStar ARMv7 SoC周围的印刷电路板上，有一些可见的测试点（test pads）。通过在相机启动时用万用表测量它们，其中一个似乎像是3.3V信号。将该引脚连接到逻辑分析仪，我们能够确认它确实是一个UART-TX引脚，波特率为115200。然后通过试错法，在同一组引脚中也找到了UART-RX引脚。我们将UART引脚连接到FTDI USB串口适配器，尽管收到了大量的日志信息，但设备似乎忽略从RX接收的任何输入，因此没能通过这种方式获得登录Shell。

## OTA固件  
  
对于小米设备来说，获取OTA固件很棘手，因为没有支持页面提供可下载的固件文件，也没有公开的固件文件列表。小米设备从小米更新服务器查询更新，并从服务器获取下一个可用的OTA更新URL。  
  
然而，URL中包含固件版本，存在一些规律。因此，尽管没有完整的设备更新日志，但这仍然可能允许人们通过暴力猜测来获取固件。  
  
在我们的情况下，如上所述，此时我们已经获得了root shell，因此可以不受OTA固件镜像的限制来满足逆向工程需求。

有效载荷使用PKCS#7填充到16字节的倍数，然后用AES-128 CBC模式加密，其中 密钥 key = MD5(令牌 token)  
， 初始化向量 iv = MD5(密钥 key + 令牌 token)  
。明文通常是JSON。这种加密确保了有效载荷的机密性（confidentiality）。  
  
校验和（checksum）的计算方法是：首先用令牌填充其字节，然后对整个数据包使用MD5。这意味着校验和充当了消息认证码（MAC, Message Authentication Code），因为计算它需要知道令牌。结合用于防止重放攻击（replay attacks）的时间戳，这个校验和构成了协议的认证机制。  
## 设置过程  
  
当摄像头首次从包装盒中取出时，它处于特殊的“出厂”状态。也可以通过按住摄像头背面的按钮10秒钟执行恢复出厂设置，随时重新进入此状态。  
  
在此状态下，摄像头会运行自己的Wi-Fi接入点，且无需认证。手机可以连接到这个接入点，并使用小米家庭应用来设置设备。  
  
  
![](https://mmbiz.qpic.cn/sz_mmbiz_png/TKdPSwEibsZgDTR8RQzEsbCqzdbfoRCoCp9W6ymkORoJQ6GiaAaSzovNWjpn7KP0iagicyPH1NxXm3JpBjHjJIyDQhKTpObRx41SMQpZ8KbQKzQ/640?wx_fmt=png&from=appmsg "")  
  
握手序列图  
  
设置过程要求手机使用miIO协议向摄像头发送一系列特定的“握手”数据包。摄像头对握手消息（R0  
）的响应包含一个十六进制编码的令牌。该令牌成为后续密钥生成步骤（K = HKDF(ECDH(s2, p1), 盐 salt)  
）中使用的salt  
，如前节描述数据包格式所示。  
  
一旦序列成功完成（包括最终的HMAC验证步骤），所有后续消息都将包含一个MAC（替换原有的ff  
或令牌），并且实际有效载荷也会被加密，如上面再次展示的那样。  
  
默认情况下，完成该序列需要一个随机的、设备特定的设置code  
。用户需要使用小米家庭应用程序扫描设备底部的二维码获取该code  
，这需要物理接触设备。另一个选项是允许设备随机生成一个一次性code  
，它会通过扬声器传达code  
，正如  
此  
教程视频所示。  
  
这个握手过程不仅验证了手机物理接触摄像头的能力，还允许手机在不暴露code  
本身的情况下，确认摄像头也知道该设置code  
。这防止了攻击者冒充摄像头欺骗应用程序或泄露code  
。椭圆曲线迪菲-赫尔曼密钥交换（Elliptic Curve Diffie-Hellman key exchange）也使得中间人攻击（Man-in-the-Middle attack）无法得逞。  
## 总结：攻击面  
  
从以上描述可以看出，miIO  
协议从密码学角度看设计似乎是合理的，考虑到了中间人攻击的场景，并且所有后续的设置后消息都通过共享密钥保护，其MAC覆盖数据包字段的方式避免了哈希扩展攻击（hash-extension attacks）。这发生在任何其他额外的身份验证可能进一步缩小攻击面之前。  
  
然而，事实证明，设计者不仅在加密实现上犯错，使设置流程本身可以被绕过，甚至在（初始）数据包处理代码中还留下了一个允许攻击者通过Wi-Fi接口实现完全远程代码执行（RCE, remote code execution）的内存破坏漏洞。  
## 漏洞  
## 漏洞 #1：小米 - miIO协议认证绕过  
  
由于握手序列设计存在缺陷，攻击者可以通过重放（replay）摄像头发送的某些值，在无需知道设置code  
的情况下完成整个设置流程。这消除了攻击者物理接触设备的必要性。

## 漏洞 #2：小米 - miIO客户端密码学弱伪随机数生成器（cryptographically weak PRNG）  
  
在握手序列中用于生成密钥对和随机前缀时，摄像头使用了uClibc库默认的加性滞后斐波那契生成器。这不适用于密码学目的。  
  
平均发送22个数据包后，所有未来的随机数都可以被可靠预测，这导致设置流程中所用密码学原语（cryptographic primitives）的要求失效。  
  
为了“生成”足够的反射随机性（reflected randomness）来实现此目的，我们可以简单地重新启动握手序列，摄像头将持续发送带有新生成的随机令牌的新R0  
响应。  
  
我们在实践中演示此漏洞利用的原理证明PoC可在  
此处  
找到。  
  
回到协议设计，值得问为什么这很重要。一般来说，伪随机数生成器（PRNG）能被轻易攻破绝不是个好兆头，但我们仍然想知道具体影响是什么。  
  
首先，考虑到基于二维码的code  
长度为128位，为了试图暴力破解code  
而攻破伪随机数生成器价值不大。此外，即使我们等到那个序列，我们总是会在收到H2  
后得到R2  
，所以如果我们想尝试这种暴力破解（考虑到我们对计算资源现实的认识，这无论如何都将是徒劳无功的），我们也不需要预测除code  
以外的任何输入。  
  
另一方面，在使用随机生成code  
的场景下，该code  
只有4位数字，因此其熵极低，甚至无需离线暴力破解即可轻易攻破。当然，鉴于设计上允许物理接触设备的人强制重置并强迫摄像头大声说出code  
，伪随机数生成器的设计缺陷似乎也不是关键问题。  
  
然而，存在第三种情况，这使得伪随机数生成器问题变得有意义。从序列中我们可以看到，一旦攻击者能够攻破伪随机数生成器，即使它不再冒充小米家庭应用程序，也没有任何东西能阻止它精确计算出后续生成的K  
值（因为所有的s2  
、p2  
、salt  
都将变得可预测）。  
  
换句话说，攻击者可以首先执行伪随机数生成器攻击，“离开”并让真正的所有者自行重新进行正确的设置，然后继续窃听miIO  
接口上的所有通信。其中，包括以明文形式推送到摄像头的Wi-Fi凭据。  
  
本质上，伪随机数生成器漏洞直接破坏了任何802.11级别的安全措施，并提供了方便的局域网直接访问权限——当然，也为任何潜在的后续攻击铺平了道路。

```
    AesKeyExpansion();
  }
  if (iv != 0x0) {

  }
  uint offset = 0;

    dst = plaintext + offset;

    AesIv = src;

    copy16(dst, src);

    AesCurrentBlock = dst;
    AesBlockDecrypt();
  }
}
```  
  
部分块的大小保存为remainder_bytes  
，但size  
变量并未减去这个量。这导致最后一个（在if条件中的）块会从输入缓冲区之外读取密文，并将解密后的明文写入到位于堆（heap）上的输出缓冲区之外。  
  
由于程序使用的是miIO  
协议头部指定的数据包长度，攻击者可以通过发送大于此长度的UDP数据包来控制输入缓冲区之后的字节，从而控制写入明文堆块边界之外的16个字节。

## 找一个可用的伪造堆块

这意味着，我们的伪造堆块为了有用，需要满足三个条件：  
- 它的大小必须是合理的，意味着它后续可以被分配。如果太小（例如0-8字节），我们将永远无法分配它。如果太大，它很可能被程序的其他部分（包括其他线程）使用，在我们之前将其填充。  
  
- 伪造堆块的前向指针必须为0，否则分配器会跟随指针查找下一个堆块，这很可能导致崩溃，因为它不符合这些条件。  
  
- 分配器必须认为该伪造堆块的前一个和后一个堆块当前正在使用中。否则，它会尝试对它们进行unlink()操作，这会导致崩溃或中止，因为前向和后向指针不正确。

uClibc中的malloc

此外，如果数据包足够大（大于1KB），分配器会可预测地将其放在堆的末尾，因为它无法放入其他堆块之间的空洞中。由于程序在启动时总是执行相同的分配，并且假设当前没有其他大的分配正在使用，我们知道数据包最终会存放的地址。  
  
这是一个非常合理的假设，因为该程序是单线程的，并且在正常操作期间不会发生内存泄漏，这意味着即使它之前接收过一个大数据包，该数据包在处理我们的数据包时也已被释放，最终仍会将我们的数据包放在堆的末尾。  
  
由于我们可以可靠地预测数据包将被写入的位置，我们可以将r3

更好的是，万一我们的假设在某种程度上不成立，利用（使用了错误地址）将导致进程崩溃，促使mi_daemon

## 植入Shellcode  
  
接下来，我们可以尝试调用二进制文件中的函数。虽然由于地址空间布局随机化我们不知道libc.so  
的地址，但我们可以利用二进制文件的程序链接表（PLT）来调用链接到它的标准库函数。  
  
这样一个函数是system()  
，我们可以用它来在设备上执行任意命令。然而，这种方式无法方便地启动反向Shell或绑定Shell，迫使我们为想要运行的每个命令重新执行利用程序。我们也无法轻松获取命令的输出。  
  
或者，我们可以使用以下调用链：  
```

fd = open("/proc/self/mem", O_RDWR);
lseek(fd, 0x13000, SEEK_SET);
write(fd, shellcode_ptr, shellcode_length);
```  
  
其中0x13000  
可以是任何可执行的任意地址。我们无需寻找代码片段来保存open()  
返回的文件描述符，而是可以多次调用open()  
，以便容易猜出对应/proc/self/mem  
的文件描述符编号。注意write()  
调用之所以有效，是因为在Linux内核的默认配置下，/proc/self/mem  
会忽略写入保护，允许我们更改可执行段。  
  
将绑定Shell的Shellcode写入0x13000后，我们的最后一步是跳转到它，完成漏洞利用。由于miio_client  
进程以root权限运行且没有任何沙箱限制，我们完全掌握了整个系统。

## 漏洞利用后阶段：云端越狱  
  
“云端越狱”的目标是修改摄像头的运行时环境，以使其根本不需要配置任何云端连接。  
  
我们基本上需要两件事：替换原本通过云端完成的功能 + 确保云端不会干扰设备。  
## 实现  
  
这通过4个步骤实现：  
- 获得设备上的root访问权限。  
  
- 获得持久性（跨越重启）。  
  
- miIO  
替换：实现与设备通信的新方式。  
  
- 通过防火墙阻断云端干扰。  
  
上面描述的漏洞利用提供了步骤1。  
  
对于步骤2，需要找到一个操作系统脚本或二进制文件，该文件会被执行并且可以在不刷写整个设备的情况下被更改。脚本/mnt/data/sysctl  
就是这样一个文件。  
  
为了运行自定义代码，通过LD_PRELOAD  
预加载我们的库有效（即使在noexec  
分区上），它可以劫持供应商提供的二进制文件。为此，我们针对imi_mike  
二进制文件，它会启动我们的植入程序（implant）。  
  
对于步骤3，越狱植入程序本身（一个约600行的“可执行”代码）会调用小米也用来移动摄像头和访问摄像头视频流的库函数。这段代码实现了通过TLS将视频流发送到服务器（使用MPEG TS  
格式），以及将内容保存到SD卡并进行非对称加密的能力。它还接受命令来移动马达。  
  
在服务器端，可以运行一个简单的基于ffplay  
命令行的解决方案，甚至可以使用一个更用户友好的设置，该系统使用一个名为frigate NVR的用于IP摄像头的Web图形用户界面。  
  
最后，对于步骤4，因为我们打算实现一组最小的更改，我们不直接进一步修补摄像头软件堆栈来防止访问云端。相反，既然此时云端对于功能来说不再是必要的，首选设置可以简单地依赖自定义的防火墙规则，以确保设备不允许任何意外的互联网访问。
